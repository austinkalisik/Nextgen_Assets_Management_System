Nextgen Assets Management System
Entity Relationship Diagram (ERD) Overview

Primary ERD References
- Visual ERD: graph.png
- Generated text ERD snapshot: wireframe/erd-current.txt
- This document explains the generated ERD in plain language

Generation Source
- graph.png was regenerated from the live local project database using:
- php artisan generate:erd graph.png --format=png --no-interaction
- Supporting text output was generated with:
- php artisan generate:erd wireframe/erd-current.txt --text-output --no-interaction

System Type
- Laravel + ReactJS CRUD application
- Laravel handles models, migrations, controllers, validation, authentication, and business rules
- ReactJS handles the dashboard, CRUD pages, forms, tables, filters, and interactive workflows

Business Ownership
- Product: Nextgen Assets Management System
- Owned by: Nextgen Technology
- Website: https://nextgenpng.net/

Important Scope Note
- The generated ERD is model-based and currently shows 10 modeled entities
- It does not include every table in the database
- For example, framework/support tables and the key/value settings table are not shown in graph.png
- Use graph.png as the generated operational model ERD, not as a complete all-tables database dump

Main Generated ERD Entities
- Asset
- AssetLog
- Assignment
- Category
- Department
- Item
- StockMovement
- Supplier

Core Functional Meaning
- User manages and receives assets, creates logs, and receives notifications
- Item is the main inventory record used by the system
- Assignment records issued and returned assets
- StockMovement records quantity changes
- AssetLog records audit activity
- Category and Supplier organize inventory
- Department groups assignments
- SystemNotification records alerts and system messages

Depreciation Note
- The generated ERD now includes the depreciation fields on Item
- This means graph.png is updated for the depreciation feature
- These fields support fixed-asset valuation and book-value calculations in the system

Generated ERD Reality Note
- Because graph.png is generated from the live local database, it reflects the database as it exists right now
- That means it may show legacy columns that are still present locally even if current application logic does not rely on them heavily

Current Legacy Columns Visible in the Generated ERD
- Assignment still shows assigned_department_id alongside department_id
- User still shows profile_photo_path alongside profile_photo

How To Interpret Those Legacy Columns
- department_id is the main assignment department field used by the current app flow
- profile_photo is the main user profile image field used by the current app flow
- The generated ERD still exposes assigned_department_id and profile_photo_path because they exist in the current local database state

What graph.png Does Not Fully Represent
- settings key/value configuration table
- Laravel support tables such as sessions and password_reset_tokens
- cache/jobs framework tables

So the correct way to describe graph.png is:
- it is the current generated ERD for the modeled operational entities
- it is not the full database inventory of every table

Operational Data Flow
1. User logs in
2. Item records are created and updated
3. StockMovement records opening stock, stock in, stock out, return, and adjustment
4. Assignment records who received an item and when
5. AssetLog records audit actions
6. SystemNotification records alerts and workflow notifications
7. Department, Category, and Supplier support organization and filtering

Documentation Alignment Rule
To keep the documentation consistent:
