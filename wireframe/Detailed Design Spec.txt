Nextgen Assets Management System
Detailed Design Spec Context

ERD Reference
- Main visual ERD: graph.png
- Supporting database explanation: wireframe/ERD - Nextgen Assets Management System.txt
- Generated ERD snapshot: wireframe/erd-current.txt

System Type
- Laravel + ReactJS CRUD application
- Laravel provides backend APIs, validation, models, controllers, authentication, and business rules
- ReactJS provides the frontend interface, page rendering, forms, tables, filters, and client-side navigation

System Structure
- Login and authentication
- Dashboard and system overview
- Inventory and stock management
- Assignments and returns
- Suppliers, categories, and departments
- Users and profile management
- Notifications and activity logs
- Settings and branding

Why This Design Spec Exists
- to make the interface clearer
- to make the system look more premium and corporate
- to keep all CRUD modules visually consistent
- to align the UI with the real structure of the application

Current ERD Alignment Notes
- graph.png now comes from the live generated ERD output
- The generated diagram is operational-model focused, not a full all-tables schema map
- Item in the ERD includes depreciation fields and remains the main working inventory entity in the application
- Some legacy database columns may still appear in the generated ERD, so UI notes should follow current business usage while not contradicting the diagram

**Global Design Rules**
- Primary goal: clarity first, premium second.
- Every page should have one clear main action.
- Important numbers should be large and high-contrast.
- Supporting text should be readable but quieter.
- Avoid repeating the system name in multiple places on the same screen.
- Use consistent spacing, card radius, shadows, and button styles.

**Navigation Structure**
Sidebar groups:
- `Operations`
  - Dashboard
  - Inventory
  - Assignments
  - Suppliers
- `Administration`
  - Categories
  - Departments
  - Users
  - Notifications
  - Activity Logs
  - Settings
  - Profile

Top bar should contain:
- global search
- notification bell
- user card
- logout

Do not repeat:
- full system name again inside every page header unless necessary

**1. Login Page Spec**
Purpose:
- establish trust
- show ownership by Nextgen Technology
- keep sign-in simple

Visual priority:
1. sign-in card
2. brand identity
3. short value statement
4. subtle animated background

Left panel content:
- logo
- product name
- ownership line
- headline
- short explanation
- 3 supporting cards:
  - security
  - shared office use
  - PNG time

Right panel content:
- sign-in title
- email field
- password field
- remember email
- submit button
- support note

Important UX rules:
- login form must remain visually calmer than the animated background
- background motion must be slow and non-distracting
- button must be the strongest object on the form

**2. Dashboard Spec**
Purpose:
- show system health instantly
- allow fast action

Visual priority:
1. main dashboard heading
2. quick actions
3. stat cards
4. recent assignments
5. right-side support panels

Header area:
- badge: `System overview`
- title: `Admin Dashboard`
- short summary sentence
- refresh button
- secondary text link: `View all assets`

Quick action row:
- Add New Asset
- Assign Asset
- Stock Operations

These should be:
- bigger than normal buttons
- visually differentiated
- easy to tap/click

Stat card row:
- total assets
- available
- assigned
- maintenance
- low stock
- overdue

Rules:
- label small
- number large
- icon visible
- color tied to meaning

Recent Assignments panel:
Columns:
- asset
- receiver
- department
- date
- status

Right panels:
- Quick Actions
- Performance / system message
- optional future card: low stock spotlight

Do not:
- overload dashboard with too many charts
- show unnecessary explanatory paragraphs

1. add item action

4. item form when opened

Filter row:

- category
- clear filters

Table columns:
- name
- asset tag
- sku
- brand
- category
- supplier
- quantity
- book value
- stock state
- date added
- actions

Inventory form sections:
1. Tracking Mode
2. Item Basics
3. Stock Details
4. Depreciation
5. Lifecycle / Identity
6. Notes

Required fields must show `*`:
- Name
- Initial Quantity
- Category
- Supplier
- Lifecycle Status

Required only when depreciation enabled:
- Unit Cost
- Useful Life

Delete UX:

Component Priority Rules
Use these visual priorities:
- primary action: solid gradient or strong filled button
- secondary action: white or soft neutral button
- destructive action: red outline/fill with confirmation modal
- metrics: large number, small uppercase label
- cards: rounded, shadowed, consistent spacing
- tables: clean headers, light hover state, readable rows

Accessibility / Clarity Rules
- dark text on light surfaces must be high contrast
- light text on dark hero panels must stay readable
- supporting text should never be too faint
- avoid duplicate labels or repeated brand lines
- form labels must be visible and not rely on placeholders only

What Step 3 Should Be
Step 3 is the implementation roadmap:
- which page to redesign first
- what shared classes/components to improve
- what to code in what order

1. login
2. dashboard
3. inventory
4. assignments
5. settings
6. users
7. notifications
8. remaining admin pages
