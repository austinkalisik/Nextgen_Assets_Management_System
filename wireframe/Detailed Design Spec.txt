Nextgen Assets Management System
Detailed Design Spec

System Type
- Laravel backend API
- ReactJS frontend
- Vite build
- Authenticated single-page admin workspace
- Current inventory model: Item
- Current people model: Receiver belongs to Department

Design Goal
- Make the app clear, professional, and hard to misuse.
- Use department-first reporting where people or assignments are involved.
- Keep forms plain and predictable.
- Keep export buttons close to the filters they obey.

Navigation Structure
Operations:
- Dashboard
- Inventory
- Assignments
- Suppliers
- User Guide

Administration:
- Categories
- Departments
- Receivers
- Users
- Notifications
- Activity Logs
- Settings
- Profile

Global Shell
- Sidebar owns navigation and signed-in user card.
- Top bar owns global search, notification bell, user card, and logout.
- Page headers should show the page purpose and one main action.
- Avoid repeating the full product name inside every page.

1. Login Page
Purpose:
- Let authorized users sign in.
- Communicate the product is owned by Nextgen Technology.

Role message:
- Admin
- Manager
- Asset Officer
- Procurement Officer
- Auditor
- Staff

2. Dashboard
Purpose:
- Show inventory and assignment health quickly.

Must show:
- Quick actions: Add New Asset, Assign Asset, Stock Operations
- Stat cards: total assets, available, assigned, maintenance, low stock, overdue
- Recent assignments with asset, receiver, department, date, status
- Quick links to common pages

3. Inventory
Purpose:
- Manage items, stock quantity, lifecycle status, depreciation, and identity fields.

Top controls:
- Search
- Add Item
- Export CSV
- Stock state filter
- Category filter
- Clear filters

Table columns:
- Name
- Asset Tag
- SKU
- Brand
- Category
- Supplier
- Quantity
- Book Value
- Stock State
- Actions

Inventory form sections:
- Tracking Mode
- Item basics
- Stock availability
- Depreciation
- Category, supplier, and identity
- Notes / description

Tracking Mode rules:
- Bulk stock: one inventory row can represent many units.
- Serialized asset: one unique asset, normally quantity 1, with serial number tracking.

Date rules:
- Purchase Date uses YYYY-MM-DD input with calendar picker button.
- Depreciation Start Date uses the same YYYY-MM-DD input with calendar picker button.
- Do not allow six-digit year entry or browser date reset behavior.

Depreciation rules:
- Track depreciation only for fixed assets that lose value over time.
- Required when enabled: useful life and depreciation start date.
- Unit cost and salvage value feed book-value calculations.

4. Assignments
Purpose:
- Give out assets, return assets, check people, check stock, and review history.

Main layout:
- Header with Refresh and Give Out
- Metrics: Issued Out Now, Returned, People Holding Assets, Total Issued Qty
- Assignment Work Area with six views:
  - Issued Out
  - Give Out
  - Returned
  - People
  - Stock Check
  - History

Department-first assignment rule:
- A receiver must belong to a department.
- The Give Out form should label the selector as Department / Person Receiving.
- Receiver options should be grouped by department.
- Reports should display department next to the person.

Report period:
- All time
- Today
- This month
- This year
- Choose dates

Export rules:
- Export CSV and Export PDF must use the current search, tab, and report-period filters.
- Choose dates uses the same stable YYYY-MM-DD calendar input pattern used by inventory.

Return UX:
- Do not use browser confirm for important actions.
- Use an in-app confirmation modal for Mark Returned.

5. Departments
Purpose:
- Group receivers and assignment reports by office/business unit.

Form:
- Department Name
- Description

Table:
- Department Name
- Description
- Total Assignments
- Active Assignments
- Actions

Delete rule:
- Do not delete a department with linked receivers or assignments.

6. Receivers
Purpose:
- Manage the people or teams who can receive assets.

Rules:
- Receiver Name is required.
- Department is required.
- Email and phone are optional.
- Active flag controls whether the receiver can be selected.

Table:
- Receiver Name
- Department
- Email
- Phone
- Active
- Assignments
- Actions

Assignment connection:
- Assignment should use receiver + department together, not a loose person name only.

7. Users
Purpose:
- Manage login accounts and role permissions.

Roles:
- Admin: full system access
- Manager: oversees operations
- Asset Officer: manages inventory and assignments
- Procurement Officer: manages supplier and stock intake work
- Auditor: read-only audit and report access
- Staff: standard user access

Important UX:
- Only administrators can manage users.
- If admin switches user, show impersonation banner and Return to Admin action.
- Non-admin impersonated view must not expose admin-only management controls.

8. Activity Logs
Purpose:
- Audit trail for asset actions, stock operations, and assignment changes.

Controls:
- Search by action, item, user, or notes
- Action filter
- Refresh
- Export CSV
- Export PDF

Export rule:
- Exports must match the current filter/search results.

Table:
- Date
- Action
- Item
- User
- Notes

9. Notifications
Purpose:
- Show system alerts and workflow messages.

Controls:
- Status filter
- Priority filter
- Type filter
- Search
- Mark all read
- Clear read

Notification list:
- Priority
- Title
- Message
- Type
- Time
- Open action

10. Suppliers and Categories
Suppliers:
- Name, email, phone, address, linked items, actions.

Categories:
- Name, description, linked items, actions.

11. Settings
Sections:
- Branding
- Organization
- Operations
- Feature Controls

Controls:
- System logo
- Officer profile image
- System name
- Tagline
- Company name
- Website
- Support email
- Low stock threshold
- Assignment overdue days
- Items per page
- Email notifications
- Maintenance alerts
- Switch user access

12. Profile
Purpose:
- Let the current user manage personal account details.

Sections:
- Profile image
- Name and email
- Account details
- Password update

Design Rules
- Primary action: filled blue/teal gradient button.
- Secondary action: neutral outlined button.
- Destructive action: red and confirmed.
- Export action: neutral button near the filters it uses.
- Date fields: visible YYYY-MM-DD text plus calendar picker.
- Tables: simple headers, readable rows, clear action buttons.
- Cards: use for repeated records and grouped form/report areas, not nested decoration.
